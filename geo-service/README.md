# geo-service

FastAPI geospatial micro-service for the ChargeWise EV charging network.
Loads city-level GeoJSON layers, scores EV charger candidate locations
across five spatial factors, and serves the results to the Go API gateway.

---

- Python 3.11+
- `rasterio` (needed only for Step 5 of the data pipeline):

  ```bash

  ```

- Two external files you must obtain manually before running the pipeline:
  - `BBMP.geojson` ‚Äî Bengaluru ward boundaries, available from
    [datameet/municipal-profiles](https://github.com/datameet/municipal-profiles)
  - `IND_ppp_2020_adj_v2.tif` ‚Äî India WorldPop raster, download from
    [worldpop.org](https://www.worldpop.org)

## Configuration

All configuration is loaded from environment variables at startup. The process
exits immediately with a structured error log if a required variable is absent.

| Variable | Required | Default | Description |
|---|---|---|---|
| `DATA_DIR` | ‚úÖ | ‚Äî | Path to the directory containing city GeoJSON folders, e.g. `data` |
| `DEFAULT_CRS_EPSG` | ‚úÖ | ‚Äî | EPSG code for the projected CRS used for all spatial ops; must be `32643` for Indian cities (UTM Zone 43N) |
| `LOG_LEVEL` | | `INFO` | One of `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` |
| `WARM_CITIES` | | `bengaluru` | Comma-separated city directory names to pre-load at startup |
| `HOST` | | `0.0.0.0` | Uvicorn bind host |
| `PORT` | | `8001` | Uvicorn bind port |

### Startup warming

On startup the service loads every city listed in `WARM_CITIES` into the
`DatasetRegistry` cache before accepting traffic. The `GET /ready` probe
returns `503` until all cities have finished loading. This moves the
cold-load penalty (~44 s per city, dominated by `population_grid.geojson`
at 195 k rows) to deploy time rather than the first HTTP request.

If warming fails for a city, a warning is logged and the service
continues ‚Äî that city will cold-load on the first request.

## Data Pipeline

Run these scripts once, **in order**, from the `geo-service/` directory.
Each step's output is the input for the next.

### Step 1 ‚Äî Fetch OSM data

```bash
python scipts/fetch_osm_data.py
```

Queries the Overpass API for Bengaluru POIs (EV chargers, fuel stations,
parking, roads, metro stations, malls, tech parks) and saves:

- Raw Overpass JSON ‚Üí `data/raw/bengaluru_overpass_raw.json`
- Converted GeoJSON ‚Üí `data/raw/bengaluru_overpass_export.geojson`

If `bengaluru_overpass_raw.json` already exists it is reused ‚Äî no repeat
API call is made.

---

### Step 2 ‚Äî Prepare ward boundaries

```bash
python scipts/fetch_ward_boundaries.py \
    --input ~/Downloads/BBMP.geojson \
    --city bengaluru \
    --data-dir data
```

Validates and reprojects the raw BBMP ward boundary file to EPSG:4326.
Fails with a clear error if the bounds look like projected metres rather
than WGS-84 degrees (guards against mislabelled CRS metadata in
community-sourced files).

**Output:** `data/bengaluru/ward_boundaries.geojson`

| Flag | Required | Default | Description |
|---|---|---|---|
| `--input` | ‚úÖ | ‚Äî | Path to the downloaded `BBMP.geojson` |
| `--city` | ‚úÖ | ‚Äî | City directory name, e.g. `bengaluru` |
| `--data-dir` | | `data` | Root data directory |

---

### Step 3 ‚Äî Compute city bounding box

```bash
python scipts/compute_city_bbox.py \
    --city bengaluru \
    --data-dir data
```

Dissolves all ward polygons into a single convex-hull bounding polygon.
Prints the `--bbox` value needed in Step 5.

**Output:** `data/bengaluru/city_bbox.geojson`

> Copy the printed bbox tuple (e.g. `77.459880,12.833490,77.784361,13.142620`)
> for use in Step 5.

| Flag | Required | Default | Description |
|---|---|---|---|
| `--city` | ‚úÖ | ‚Äî | City directory name |
| `--data-dir` | | `data` | Root data directory |

---

### Step 4 ‚Äî Split OSM layers

```bash
python scipts/split_osm_layers.py \
    --input data/raw/bengaluru_overpass_export.geojson \
    --city bengaluru \
    --data-dir data
```

Filters the bulk GeoJSON export from Step 1 into seven per-layer files.
Empty layers are written as empty GeoJSON files (not omitted) so the
`DatasetRegistry` triggers the correct missing-layer warning path rather
than a file-not-found error.

**Output (`data/bengaluru/`):**

| File | Content |
|---|---|
| `ev_chargers.geojson` | EV charging stations |
| `fuel_stations.geojson` | Petrol / CNG stations |
| `parking.geojson` | Parking areas |
| `roads.geojson` | Motorway / trunk / primary / secondary roads |
| `metro_stations.geojson` | Metro / subway stations |
| `malls.geojson` | Shopping malls |
| `tech_parks.geojson` | IT and tech parks |

| Flag | Required | Default | Description |
|---|---|---|---|
| `--input` | ‚úÖ | ‚Äî | Path to the bulk OSM GeoJSON export |
| `--city` | ‚úÖ | ‚Äî | City directory name |
| `--data-dir` | | `data` | Root data directory |

---

### Step 5 ‚Äî Build population grid

```bash
python scipts/build_population_grid.py \
    --raster ~/Downloads/IND_ppp_2020_adj_v2.tif \
    --bbox 77.459880,12.833490,77.784361,13.142620 \
    --city bengaluru \
    --data-dir data \
    --buffer-km 3
```

Clips the India-wide WorldPop raster to the city bbox, vectorises populated
pixels into GeoJSON polygons, and writes the result. Nodata and zero-
population cells are discarded to keep file size manageable.

Use the `--bbox` value printed at the end of Step 3.

**Output:** `data/bengaluru/population_grid.geojson`

| Flag | Required | Default | Description |
|---|---|---|---|
| `--raster` | ‚úÖ | ‚Äî | Path to `IND_ppp_2020_adj_v2.tif` |
| `--bbox` | ‚úÖ | ‚Äî | `minx,miny,maxx,maxy` in WGS-84 degrees |
| `--city` | ‚úÖ | ‚Äî | City directory name |
| `--data-dir` | | `data` | Root data directory |
| `--buffer-km` | | `3.0` | Padding around the bbox in km so edge-of-city candidates don't undercount population |

## API Reference

Interactive Swagger docs: `http://localhost:8001/docs`

### `GET /health`

Liveness probe ‚Äî returns `200` as soon as the process is running.
Use for Kubernetes `livenessProbe` (restart on failure). Do **not** use
to gate traffic ‚Äî use `/ready` instead.

```json
{ "status": "ok", "service": "geo-service" }
```

---

### `GET /ready`

Readiness probe ‚Äî returns `200` only after all `WARM_CITIES` datasets have
finished loading. Returns `503` while warming is in progress.

Use for Kubernetes `readinessProbe` and load-balancer health checks.

**200 OK:**

```json
{
  "status": "ready",
  "service": "geo-service",
  "warmed_cities": ["bengaluru"]
}
```

**503 Service Unavailable:**

```json
{
  "status": "warming",
  "service": "geo-service",
  "warmed_cities": [],
  "detail": "Dataset cache warming in progress ‚Äî retry in a few seconds."
}
```

### `GET /analysis?city=&chargerType=`

Returns score distribution statistics and per-ward breakdowns for a city.

| Query param | Required | Description |
|---|---|---|
| `city` | ‚úÖ | Supported city name |
| `chargerType` | ‚úÖ | `SLOW`, `FAST`, or `DC_FAST` |

**200 OK:**

```json
{
  "city": "Bengaluru",
  "chargerType": "FAST",
  "total_candidates": 312,
  "score_mean": 54.3,
  "score_median": 52.0,
  "score_p90": 78.1,
  "coverage_pct": 91.4,
  "ward_stats": [
    { "ward_name": "Koramangala", "candidate_count": 18, "mean_score": 71.2 }
  ]
}
```

`coverage_pct` is the percentage of the city bounding polygon covered by
at least one candidate within 500 m.

**422 Unprocessable Entity** ‚Äî unsupported city or charger type.

---

### `GET /chargers?city=`

Returns all existing EV charger locations for a city as a GeoJSON
`FeatureCollection` (WGS-84).

**422 Unprocessable Entity** ‚Äî unsupported city.

---

### `POST /validate`

Validates a GeoJSON `FeatureCollection` (max 50 MB). Checks for null
geometry, self-intersection, and coordinates outside WGS-84 bounds.

**200 OK:**

```json
{
  "record_count": 142,
  "crs": "EPSG:4326",
  "geometry_types": ["Point"],
  "validation_errors": [
    { "feature_index": 7, "message": "Null geometry." }
  ]
}
```

**400 Bad Request** ‚Äî body is not valid JSON or not a `FeatureCollection`.

## Scoring Algorithm

The `Scorer` class computes a weighted 0‚Äì100 score for each candidate
location across five geospatial factors. All spatial operations use
GeoPandas vectorised batch methods ‚Äî no row-by-row Python loops.

| Factor | Weight | Method |
|---|---|---|
| Population density | 35% | Sum population within fixed 1 000 m buffer; normalise by 50 000 |
| Charger distance | 25% | Distance to nearest existing charger within `radius`; farther = higher score |
| Road proximity | 15% | Binary: 100 if arterial road within 200 m, else 0 |
| Parking availability | 15% | Binary: 100 if candidate intersects a parking polygon, else 0 |
| Mall proximity | 10% | Binary: 100 if mall within 500 m, else 0 |

The pipeline runs all five factors in a single `score_batch()` call that
returns factor scores and raw detail fields together, so no spatial
operation is repeated.

### Candidate generation

1. **Primary path** ‚Äî centroid of every parking polygon.
2. **Fallback** ‚Äî deterministic uniform grid at 500 m spacing across the
   city bounding box (triggered when the parking layer is empty). The
   grid uses `numpy.meshgrid` + `geopandas.points_from_xy` ‚Äî no Python-
   level loop.

### Missing layer handling

When a spatial layer is absent or fails to parse, the corresponding
factor returns 0 (or 100 for `charger_distance`, meaning maximally
under-served). The layer name is recorded in `CityDatasets.missing_layers`
and surfaced as a warning in the recommendation response.

### CRS strategy

Files on disk are WGS-84 (EPSG:4326). The `DatasetRegistry` reprojects
every layer to EPSG:32643 (UTM Zone 43N, metres) at load time so all
downstream spatial operations use metric distances. Responses are
re-projected back to WGS-84 before serialisation.

## Observability

Every request emits a structured JSON log line:

```json
{
  "timestamp": "2025-07-14T09:12:34",
  "level": "INFO",
  "logger": "app.main",
  "message": "request completed",
  "correlation_id": "3f2a‚Ä¶",
  "method": "POST",
  "path": "/recommendation",
  "status_code": 200,

```

The `X-Correlation-ID` header is propagated from incoming requests (or
generated as a UUID if absent) and echoed on every response and log entry.
